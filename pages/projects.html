Phase 1 — Environment Setup
STEP 01 // VirtualBox Installation
WHAT: Installed Oracle VirtualBox 7.2.6 on Windows 11 — our hypervisor software for hosting virtual machines. WHY: A Type 2 hypervisor lets you run multiple isolated operating systems on one physical machine. This mirrors how enterprise IT works at scale — companies run VMs on physical servers using VMware ESXi or Hyper-V. NOTE: VirtualBox is free and open source. Enterprise equivalents: VMware ESXi (Type 1), Microsoft Hyper-V (Type 1). Type 1 runs on bare metal, Type 2 runs on top of a host OS.
// Oracle VirtualBox 7.2.6 — installed and booting first VM
STEP 02 // Windows Server 2022 ISO
WHAT: Downloaded the Windows Server 2022 180-day evaluation ISO (4.7GB) from Microsoft Evaluation Center. WHY: The ISO is the installation image for our server OS. Windows Server 2022 is the current enterprise standard — what most IT environments run today. NOTE: Free 180-day eval from Microsoft — legitimate and widely used for lab/learning. The .iso mounts as a virtual DVD drive in VirtualBox.
STEP 03 // VM Creation & Hardware Config
WHAT: Created a new VM named 'Windows Server 2022' with 4GB RAM, 2 CPUs, and 50GB virtual hard disk. Attached the Windows Server ISO as the boot media. WHY: Configuring VM hardware is like speccing out a PC before buying it. We gave it enough resources to run Windows Server comfortably without impacting the gaming rig. NOTE: 4GB RAM = comfortable for a DC. 2 CPUs = smooth performance. 50GB disk = plenty for Server + AD. Host machine retains remaining 28GB RAM for gaming.
// VM configured: 4096MB RAM · 2 CPUs · 50GB disk · ISO attached
COMPLETE
Phase 2 — Windows Server 2022 Installation
STEP 04 // Boot from ISO
WHAT: Booted the VM from the Windows Server 2022 ISO — the Microsoft Server Operating System Setup wizard launched. WHY: Booting from an ISO in VirtualBox is identical to booting a physical PC from a USB install drive. The VM's BIOS reads the ISO and launches the installer. NOTE: If prompted 'Press any key to boot from CD/DVD' you must click inside the VM window and press a key quickly — it times out fast.
// Microsoft Server Operating System Setup — ready to install
STEP 05 // OS Edition Selection
WHAT: Selected 'Windows Server 2022 Standard Evaluation (Desktop Experience)' — the GUI version of Server 2022. WHY: Server comes in two UI modes: Server Core (no GUI, command line only — used in production for security and efficiency) and Desktop Experience (full GUI — better for learning). We chose Desktop Experience to see every step visually. NOTE: In production environments, Server Core is preferred because less software = smaller attack surface = more secure. For learning, Desktop Experience is the right choice.
// Selected: Windows Server 2022 Standard Evaluation (Desktop Experience)
STEP 06 // Custom Installation
WHAT: Selected 'Custom: Install Microsoft Server Operating System only' — a fresh clean install. WHY: Custom = fresh install from scratch. Upgrade = migrate from an older Windows Server version. Since our virtual disk is empty, Custom is the only valid option. NOTE: This is the standard choice for any new server deployment. IT professionals always do clean installs on new hardware rather than upgrading from older systems when possible.
// Custom install selected — fresh deployment on empty virtual disk
STEP 07 // Drive Selection & Installation
WHAT: Selected Drive 0 (50GB unallocated virtual disk) as the install target. Windows automatically partitioned and formatted the drive. WHY: Unallocated space = blank disk with no partition or file system. Windows Setup creates a small system reserved partition and the main C: drive automatically. NOTE: Same process as installing Windows on a brand new physical hard drive. The system reserved partition stores boot files — Windows needs this separate from C: to boot correctly.
// Drive 0: 50GB unallocated — installation target selected
STEP 08 // Administrator Password
WHAT: Set the built-in Administrator account password meeting Windows Server complexity requirements. WHY: The Administrator account is the superuser on Windows Server — full control over everything. Complex password requirements exist because weak admin passwords are one of the most common enterprise security vulnerabilities. NOTE: Password requirements: 8+ chars, uppercase, lowercase, number, special character. Write it down — losing the Administrator password on a DC requires complex recovery procedures.
// Administrator password configured — complexity requirements met

Phase 3 — Active Directory & Domain Controller
STEP 09 // Install AD DS Role
WHAT: Used Add Roles and Features Wizard to install the Active Directory Domain Services role along with Group Policy Management, Remote Server Administration Tools, and AD DS management tools. WHY: Installing AD DS adds the software framework for Active Directory but doesn't activate it yet — like installing an engine before starting it. The role installation also pulls in DNS Server automatically since AD requires DNS to function. NOTE: Key components installed: AD DS role, Group Policy Management console, Active Directory Administrative Center, AD module for PowerShell. These are the tools admins use daily to manage enterprise environments.
// Active Directory Domain Services role selected for installation
STEP 10 // AD DS Installation Complete
WHAT: AD DS role installed successfully. Server Manager displayed the 'Promote this server to a domain controller' link. WHY: Installing the role is step 1. Promotion is step 2 — it actually configures AD and creates the domain. Without promotion, AD DS is installed but inactive. NOTE: This two-step process exists by design — you can install the role ahead of time and promote later. In enterprise deployments, roles are often staged before promotion during maintenance windows.
// AD DS installation complete — ready to promote to Domain Controller
STEP 11 // Domain Promotion — New Forest
WHAT: Ran the AD DS Configuration Wizard. Selected 'Add a new forest' and set the root domain name to padilla.local. WHY: Creating a new forest = building an Active Directory environment from scratch. The forest is the top-level container — it holds domains, which hold users and computers. Since we have nothing existing, we start a new forest. NOTE: The .local suffix is standard for private internal domains not registered on the public internet. In enterprise: company.local, corp.company.com, etc. Domain names are permanent — choose carefully in production.
// New forest created: padilla.local — the domain identity is set
STEP 12 // Domain Controller Options
WHAT: Configured DNS Server (auto-selected, required), Global Catalog (auto-selected), and set the Directory Services Restore Mode (DSRM) password. WHY: DNS is required because AD uses it for everything — computers query DNS to find the Domain Controller. Global Catalog means this DC stores a full copy of all AD objects. DSRM is emergency recovery mode. NOTE: DSRM password = your emergency key if AD gets corrupted. Treat it like a root password. In production, this is documented in a secure password vault. Global Catalog is mandatory on the first DC in a forest.
// DC options configured: DNS ✓ · Global Catalog ✓ · DSRM password set
STEP 13 // Prerequisites Check Passed
WHAT: All prerequisite checks passed successfully. Two warnings noted: no static IP assigned (best practice for production) and DNS delegation not available (expected for private domain). WHY: Prerequisites check verifies the server meets all requirements before making permanent changes. Yellow warnings are informational — they don't block installation but represent best practices for production environments. NOTE: Static IP warning: DCs should have static IPs in production so clients always find them. We'll set this up in a future step. DNS delegation warning: expected since padilla.local isn't registered on the internet.
// All prerequisite checks passed — cleared for domain controller promotion
STEP 14 // Server Promoted to Domain Controller
WHAT: Server promotion completed successfully. The VM rebooted automatically and came back up as the Domain Controller for padilla.local. WHY: Promotion is a one-way operation that fundamentally changes the server's role. After promotion: AD database created, DNS configured for padilla.local, SYSVOL initialized, server becomes authentication authority for the domain. NOTE: This is the moment the server became a Domain Controller — the brain of the enterprise network. Every user login, computer authentication, and policy enforcement now flows through this server.
// Promotion complete — This server was successfully configured as a domain controller
// Server Manager post-promotion: AD DS ✓ · DNS ✓ · Roles: 3 — Domain Controller for padilla.local is live

Phase 4 — Users, OUs & Group Policy
STEP 15 // Open Active Directory Users and Computers
WHAT: Launched Active Directory Users and Computers (ADUC) from Windows Administrative Tools. The padilla.local domain appeared in the left panel with default containers. WHY: ADUC is the primary GUI tool for managing everything in Active Directory — users, computers, groups, and OUs. IT admins use this tool daily for tasks like creating accounts, resetting passwords, and organizing the domain. NOTE: ADUC shows the entire domain structure in a tree view. Default containers (Builtin, Computers, Users, Domain Controllers) are created automatically when the domain is promoted. Custom OUs are added by the administrator.
// Active Directory Users and Computers — padilla.local domain structure visible
STEP 16 // Expand Domain & View Default Containers
WHAT: Expanded padilla.local to reveal the default AD containers: Builtin, Computers, Domain Controllers, ForeignSecurityPrincipals, Managed Service Accounts, and Users. WHY: Understanding the default container structure is fundamental to AD administration. Each container has a specific purpose — Domain Controllers holds DC accounts, Computers holds machine accounts, Users holds default user and group objects. NOTE: Note that default containers (like Computers and Users) are NOT OUs — they're containers and cannot have Group Policy linked to them directly. That's one of the main reasons we create custom OUs.
// Default AD containers revealed — Builtin, Computers, Domain Controllers, Users
STEP 17 // Create Organizational Units
WHAT: Created four custom OUs directly under padilla.local: _ADMINS, _USERS, _COMPUTERS, and _GROUPS. Used underscore prefix convention so they sort above default containers. WHY: OUs are the organizational backbone of Active Directory. They let you apply different Group Policies to different groups of users and computers. The underscore prefix is a common IT convention for custom OUs. NOTE: 'Protect container from accidental deletion' was checked on all OUs — this prevents catastrophic mistakes. In production, accidentally deleting an OU containing hundreds of accounts would be a major incident.
// Four custom OUs created: _ADMINS · _USERS · _COMPUTERS · _GROUPS
STEP 18 // Create Admin User Account — k.padilla
WHAT: Created user account Kristian Padilla (k.padilla@padilla.local) in the _ADMINS OU. Set password with 'never expires' for lab convenience. Added account to Domain Admins security group. WHY: Best practice is to have a separate named admin account rather than using the built-in Administrator. Named accounts create an audit trail — you can see who made changes and when. The built-in Administrator is kept as an emergency fallback only. NOTE: Two login formats auto-generated: UPN format (k.padilla@padilla.local) and legacy NetBIOS format (PADILLA\k.padilla). Both work for authentication. Enterprise naming conventions typically use firstname.lastname or flastname format.
// k.padilla confirmed as member of Domain Admins — full domain admin privileges granted
STEP 19 // Create Standard User Accounts
WHAT: Created two standard user accounts in _USERS: John Smith (j.smith@padilla.local) and Jane Doe (j.doe@padilla.local). Both set to require password change at next logon. WHY: Standard user accounts represent regular employees with no admin rights — least privilege principle. Requiring password change at first logon ensures users set their own private password that IT never knows. NOTE: Least privilege is a core security principle: users only get the access they need to do their job. j.smith and j.doe have no admin rights — they can log in and use domain resources but cannot make system changes.
// Standard users created: John Smith and Jane Doe in _USERS OU
STEP 20 // Create Security Group — IT_Admins
WHAT: Created a Global Security group named IT_Admins in the _GROUPS OU. Added k.padilla as a member. WHY: Security groups are how permissions are managed at scale. Instead of assigning permissions to individual users, you assign to a group. Need to give 50 people access to a shared folder? Give the group access — done. NOTE: Global scope = can be used across domains in the same forest. Security type = can be assigned permissions to resources. This is the most common group type in enterprise AD environments.
// IT_Admins security group created in _GROUPS — k.padilla added as member
STEP 21 // Open Group Policy Management
WHAT: Launched Group Policy Management console from Windows Administrative Tools. Expanded the forest to reveal padilla.local domain with all custom OUs and the Default Domain Policy. WHY: Group Policy Management is where administrators create, link, and manage GPOs across the entire domain. All OUs are visible here — this is the central control panel for enforcing settings across the network. NOTE: The Default Domain Policy is a built-in GPO that applies to all users and computers in the domain. It controls baseline settings like password policies and account lockout. We never edit this directly — we create our own GPOs instead.
// Group Policy Management — padilla.local forest showing all OUs and Default Domain Policy
STEP 22 // Create & Link Desktop Wallpaper GPO
WHAT: Right-clicked _USERS OU → Created a new GPO named 'Desktop-Wallpaper-Policy' linked directly to _USERS. Opened the Group Policy Editor to configure the wallpaper setting. WHY: Linking the GPO to _USERS means it automatically applies to every user in that OU — John Smith and Jane Doe. Add more users to _USERS later and they get the policy automatically, no manual configuration needed. NOTE: GPOs are applied in order: Local → Site → Domain → OU. OU-level GPOs have the highest precedence and override higher-level settings. This allows fine-grained control — different OUs can have completely different policies.
// Desktop-Wallpaper-Policy GPO created and linked to _USERS — Group Policy Editor open
STEP 23 // Configure Desktop Wallpaper GPO Setting
WHAT: Navigated to User Configuration → Policies → Administrative Templates → Desktop → Desktop. Enabled the Desktop Wallpaper setting with path C:\Windows\Web\Wallpaper\Windows\img0.jpg and Fill style. WHY: User Configuration settings follow the user to any domain machine they log into. Using an existing Windows wallpaper path means no custom file needed. This proves Group Policy is working end to end when tested on the client VM in Phase 5. NOTE: GPOs don't apply to the DC itself — they apply to domain-joined clients. The real test is Phase 5 when Windows 11 joins the domain and j.smith logs in and gets the wallpaper automatically.
// User Configuration — policies follow the user across any domain machine
// Policies expanded — Administrative Templates contains registry-based settings
// Administrative Templates — Control Panel, Desktop, Network, System categories
// Desktop folder — Desktop Wallpaper setting enabled with Windows default wallpaper path
STEP 24 // Enable Desktop Wallpaper Policy
WHAT: Inside the Group Policy Editor navigated to User Configuration → Policies → Administrative Templates → Desktop → Desktop. Located the "Desktop Wallpaper" setting, double-clicked it, set it to Enabled, entered the path C:\Windows\Web\Wallpaper\Windows\img0.jpg and set style to Fill. WHY: Enabling this setting forces the wallpaper onto any user in the _USERS OU when they log into a domain-joined machine. Using an existing Windows system wallpaper means no custom file needs to exist on client machines — it's already there on every Windows installation. NOTE: This policy won't visibly change anything on the Domain Controller itself. The real test is Phase 5 — when a Windows 11 client joins padilla.local and j.smith logs in, this wallpaper will automatically apply proving the entire AD Group Policy infrastructure works end to end. Phase 4 is now complete.

Phase 5 — Windows 11 Client VM & Domain Join
STEP 25 // Create Windows 11 Client VM
WHAT: Created a new VirtualBox VM named WIN11-CLIENT — 4096MB RAM, 2 CPUs, 50GB VDI disk, Windows 11 25H2 ISO attached. Unattended installation disabled for manual control. WHY: A separate client VM is required to simulate a real workstation joining the domain. The server VM is the Domain Controller — it should never double as a client. This mirrors real enterprise architecture where servers and endpoints are distinct machines. NOTE: Windows 11 25H2 requires EFI and TPM 2.0 — both enabled by VirtualBox automatically when Windows 11 is selected as the OS type.
// WIN11-CLIENT VM configured — 4GB RAM, 2 CPUs, Win11 25H2 ISO attached, unattended install disabled
STEP 26 // Configure Internal Network — ad-lab
WHAT: Changed the network adapter on both WIN11-CLIENT and Windows Server 2022 from NAT to Internal Network, named ad-lab. Both VMs now share the same isolated virtual network. WHY: NAT gives VMs internet access but isolates them from each other — the client couldn't see the Domain Controller on NAT. Internal Network creates a private switch between the two VMs so they can communicate directly, exactly like machines on the same LAN segment. The client needs to reach the DC's DNS to resolve padilla.local and complete domain join. NOTE: The network name ad-lab must match exactly on both VMs — VirtualBox uses the name to determine which VMs share the same virtual switch. A typo means they're on different networks and can't talk.
// WIN11-CLIENT network adapter — Internal Network ad-lab confirmed
